# CL3O — Composite Lifting Surface Structural Sizing & Optimization

CL3O sizes and optimizes the composite structure of an aircraft lifting surface
(a wing). It models the wing cross-section as **three closed cells** and
evaluates each candidate design through a pipeline of **structural idealization
→ matrix structural analysis (Euler–Bernoulli beam FEA) → Tsai–Wu failure
criteria**, then drives the design toward minimum structural mass with a
**Differential Evolution (DE)** optimizer. A companion **FastAPI + React** app
visualizes archived optimization runs.

The reference design used throughout the codebase is the **DA62** wing with the
**Wortmann FX 63-137** airfoil.

This project began as an undergraduate thesis (*Desenvolvimento de software para
dimensionamento estrutural preliminar de asas em material composto*) in
Aerospace Engineering at the **Universidade Federal de Minas Gerais
(UFMG)**, Belo Horizonte, 2026.

## How it works

A single design vector encodes, per spanwise control point, the spar positions,
flange widths, and the laminate selected for each structural role (skins, webs,
flanges). For every DE candidate the evaluator runs a 11-step pipeline:

1. Validate and decode the design vector into per-control-point variables.
2. Build the three-cell cross-section geometry, based on design vector and
   airfoil points, using structural idealization methods.
3. Assemble the global beam mesh and stiffness matrix.
4. Solve the linear static system `{F} = [K]{d}` - for every dof.
5. Recover reaction forces and internal forces per element-ends.
6. Recover boom axial and panel shear stresses.
7. Evaluate Tsai–Wu ply-by-ply margins of safety.
8. Evaluate displacement (deflection / twist) margins.
9. Compute a penalty `P(X)` from the violations.
10. Compute the structural mass `m(X)`.
11. Combine into the scalar fitness `z(X) = w_m · m(X) + P(X)`.

## Installation

CL3O is a `pip`-installable package using a `src/` layout. Python ≥ 3.10 is
required. From a clean virtual environment:

```bash
# Core scientific stack only:
pip install -e .

# Core + the visualization UI backend:
pip install -e ".[ui]"

# Add test tooling and the optional desktop launcher:
pip install -e ".[ui,dev,desktop]"

# Add post-hoc analysis tools (seaborn, required for tools/stats.py):
pip install -e ".[analysis]"
```

Core dependencies (`numpy`, `scipy`, `matplotlib`, `vtk`, `pandas`, `openpyxl`)
are declared in `pyproject.toml`; `requirements.txt` is a thin pointer to the
`ui` extra.

## Running the optimizer

The input databases under `data/` are the single source of truth and are
already populated for the DA62 reference. Run the full pipeline with:

```bash
python -m cl3o.main
```

Run configuration (aircraft, optimization name, DE hyper-parameters, the
materials and airfoils loaded) lives in the `__main__` block of
`src/cl3o/main.py` — edit it there before running. Results land in
`outputs/<aircraft>_<opt>/` (gitignored), with one pickled `RuntimeData`
snapshot per generation plus a `manifest.json`.

### Rebuilding the input databases

Each JSON database is (re)generated by running its module's `__main__` block;
parameters are hardcoded there, so edit before running:

```bash
python -m cl3o.geometry.wing                 # data/wings/
python -m cl3o.geometry.airfoil              # data/airfoils/
python -m cl3o.materials.composite_library   # data/materials/ (plies + laminates)
python -m cl3o.fea.loads.load_mapper         # data/loads/
python -m cl3o.utils.oppoints                # data/oppoints/
```

## Visualization UI

The UI reads the per-generation snapshots written under `outputs/`.

```bash
# Build the React SPA (from src/cl3o/ui/frontend/):
npm install
npm run build

# Serve the JSON API + the built SPA on one port (from the repo root):
python -m uvicorn cl3o.ui.backend.app:app --port 8000

# Or launch the standalone desktop window (requires the [desktop] extra):
python -m cl3o.ui.desktop
```

For frontend development, `npm run dev` starts the Vite dev server, which
proxies API calls to the backend.

## Analysis tools

Standalone scripts under `tools/` perform post-hoc analysis on completed runs.
Requires `pip install -e ".[analysis]"`. Outputs land in `tools/output/`.

```bash
python tools/stats.py               # LHS sampling stats, ANOVA, convergence plots
python tools/sensitivity_analysis.py  # Sobol / ANOVA sensitivity indices
python tools/tune_de.py             # DE hyper-parameter sweep via LHS
python tools/tune_penalty.py        # Penalty weight exploration
python tools/compare_runs.py        # Side-by-side force / info series across runs
python tools/profile_gen.py         # Runtime profiling
```

## Tests

```bash
pytest -m "not slow and not integration"               # fast unit tests
pytest                                                 # full suite (real DA62 session)
python -m tests.validation.validate_runtime_pipeline   # end-to-end smoke gate
```

Heavy tests are marked `slow`; tests that require real JSON database files on
disk are marked `integration`. The plot-based validators under
`tests/validation/` run headless by default (`_SHOW_PLOTS = False`); set that
flag to `True` to display figures interactively.

Golden-output regression tests (`tests/test_regression_golden.py`) pin a single
DA62 evaluation and a short DE run against frozen fixtures in `tests/golden/`,
catching unintended numerical drift.

## Notes

- Pickled snapshots in `outputs/` store fully-qualified module paths, so
  snapshots from an older package layout cannot be reloaded after a rename.
  Since `outputs/` is gitignored and regenerated per run, this is harmless.
- Units: lengths in **mm**, coordinate convention X = chord, Y = span,
  Z = vertical.
